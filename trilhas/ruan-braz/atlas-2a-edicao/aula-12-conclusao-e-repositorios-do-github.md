# Conclusão e Repositórios do GitHub

**Tempo estimado de leitura:** 6 minutos

## Objetivos de aprendizado

Ao final desta aula, você será capaz de:

- Reconhecer o GitHub como solução para versionar e compartilhar os projetos criados com IA
- Estruturar repositórios separados para cada projeto ao invés de um único monorepo
- Aplicar o GitHub Desktop como ponto de entrada para quem está começando com versionamento
- Identificar a convergência entre produto, design e engenharia como o contexto onde as ferramentas ensinadas fazem sentido

## O que foi construído e o resultado final

O Brand System está pronto. O sistema criado durante a aula é responsivo: funciona em telas grandes, médias e no celular. Esse cuidado com responsividade veio automaticamente dos agentes de design e desenvolvimento configurados, que já incluem esse critério nas entregas por padrão.

O que impressiona não é só o resultado visual. É o volume de trabalho que foi realizado em um fim de semana por uma equipe de agentes orquestrados. O sistema tem uma interface navegável com sidebar, conteúdo renderizado a partir de arquivos Markdown, e toda a estrutura de diretórios organizada. O passo seguinte, não construído durante o Atlas mas apresentado como possibilidade, é adicionar login e senha, conectar aos assets de marca, criar banco vetorial e evoluir para uma marca viva com IA.

## A convergência entre produto, design e engenharia

Antes de explicar o GitHub, a aula apresenta um conceito que contextualiza tudo que foi ensinado: o PM Playbook para IA exponencial. O ponto central é que a separação tradicional entre as áreas de produto, design e engenharia está diminuindo.

No modelo antigo, o fluxo era linear: produto define, design cria, engenharia implementa. Com IA, essas três áreas convergem para o mesmo ponto. Designers aprendem desenvolvimento. Engenheiros aprendem design e produto. Profissionais de produto aprendem engenharia. Todos estão se tornando criadores de soluções.

A Overlens chama isso de era dos criadores: um momento em que as ferramentas disponíveis permitem que uma pessoa com clareza de propósito construa coisas que antes exigiam equipes inteiras. As ferramentas ensinadas no Atlas existem dentro dessa lógica. Não são fins em si mesmas. São amplificadores de quem sabe o que quer criar.

## Por que os projetos precisam ir para o GitHub

Todos os projetos criados durante o Atlas estão salvos apenas localmente. Se o computador falhar, se um arquivo for deletado por acidente ou se o Claude Code sobrescrever algo que não deveria, não há versão anterior para recuperar. Git e GitHub resolvem isso.

Git é o sistema de versionamento instalado localmente. GitHub é onde o histórico do Git vai para a nuvem. Com o projeto no GitHub, é possível recuperar qualquer versão anterior, trabalhar em colaboração com outras pessoas e compartilhar o projeto para que outros possam baixá-lo e usar.

Os projetos criados no Atlas são três e merecem repositórios separados: BM Overlens (modelo de negócio), Brand System Overlens e Projeto Alpha (gerador de imagens). Uma pasta "Atlas 2" será criada para agrupar todos eles antes do upload.

## GitHub Desktop: a forma mais simples de começar

Para quem está começando com GitHub, a forma mais acessível é o GitHub Desktop, um programa que oferece uma interface visual para criar repositórios, fazer commits e sincronizar com o GitHub sem precisar de comandos de terminal.

O acesso é direto: busque "GitHub Desktop" no Google, clique em "Download GitHub Desktop" e instale o programa. Após fazer login com a conta do GitHub, o programa lista os repositórios existentes e permite criar novos com alguns cliques.

O que não será ensinado nesta aula por ser técnico demais para o momento: merge, branch, commit, push e deploy. Esses conceitos existem e são importantes, mas exigem mais contexto do que o tempo disponível permite. Para começar, o caminho mais simples é criar o repositório, subir os arquivos e, quando precisar de uma versão, baixar o zip do GitHub.

## Criando o repositório e subindo os projetos

*Para ver a demonstração desta seção, assista a partir de 05:52 no vídeo.*

Com o GitHub Desktop aberto e conectado à conta, o processo é criar um novo repositório, apontar para a pasta local do projeto e publicar no GitHub. Para os três projetos do Atlas, esse processo é repetido três vezes.

A diferença importante: cada projeto tem seu próprio repositório. Não existe um repositório único para tudo. Isso mantém cada projeto isolado, com histórico próprio, e facilita o gerenciamento individual de cada entrega.

Após o upload, qualquer pessoa com acesso ao repositório pode baixar os arquivos como zip e usar localmente, sem precisar entender de Git. Esse é o ponto de partida mais simples e suficiente para compartilhar o que foi construído.

## O que continua depois do Atlas

O Atlas foi construído para mostrar o que é possível fazer com IA no nível avançado em um fim de semana. O que foi entregue é um mapa, não o destino. A parte que depende de cada pessoa: praticar, refinar os agentes, aprofundar o entendimento de Git e GitHub, escolher em qual projeto focar, e ouvir os problemas reais das pessoas ao redor antes de sair construindo soluções.

A recomendação final é direta: escute mais do que fala. As ferramentas aprendidas amplificam o que você já sabe e o que você se propõe a fazer. Sem clareza de direção, a capacidade de construir rápido pode se tornar a capacidade de construir muito na direção errada.

## Coloque em prática

- Crie uma conta gratuita no GitHub em github.com.
- Baixe o GitHub Desktop, faça login e crie um repositório para cada projeto que você construiu durante o Atlas.
- Para compartilhar os projetos com outras pessoas, envie o link do repositório e instrua a pessoa a baixar o arquivo zip diretamente pelo botão disponível na interface do GitHub.
